# Chapter 5 — Protocol Reference

Wire format, bridge protocol, JNI interface, and protobuf schema reference.

## Envelope v3

All DSM wire messages are wrapped in Envelope v3 containers. This is the sole wire container — no other versions are supported.

### Framing

Every Envelope v3 message begins with a `0x03` framing byte:

```
[0x03][protobuf-encoded Envelope payload]
```

When receiving a message (e.g., a genesis response), you **must** strip the framing byte before parsing the protobuf:

```typescript
// Correct: strip framing byte
function decodeFramedEnvelopeV3(data: Uint8Array): Envelope {
    if (data[0] !== 0x03) throw new Error("Not Envelope v3");
    return Envelope.fromBinary(data.slice(1));
}

// WRONG: will fail with protobuf decode error
Envelope.fromBinary(data);
```

### Envelope Structure

The canonical Envelope definition lives in `proto/dsm_app.proto`:

```protobuf
message Envelope {
    uint32 version = 1;        // always 3
    Hash32 sender = 2;         // sender device_id
    Hash32 receiver = 3;       // receiver device_id (optional)
    bytes payload = 4;         // operation-specific protobuf bytes
    bytes signature = 5;       // SPHINCS+ signature over canonical bytes
    uint64 sequence = 6;       // monotonic sequence counter
}
```

## Bridge Binary Protocol

The React frontend communicates with the Kotlin bridge via a binary MessagePort channel.

### Message Format

```
[8-byte msgId (u64 big-endian)][BridgeRpcRequest protobuf bytes]
```

The `msgId` is a unique identifier generated by the frontend for correlating requests with responses. The bridge returns a response with the same `msgId`.

### BridgeRpcRequest / BridgeRpcResponse

```protobuf
message BridgeRpcRequest {
    string method = 1;         // RPC method name (e.g., "getBalance")
    bytes payload = 2;         // method-specific protobuf arguments
}

message BridgeRpcResponse {
    bool success = 1;
    bytes result = 2;          // method-specific protobuf result
    string error = 3;          // error message if !success
}
```

### Request/Response Flow

```
React (TypeScript)                    Kotlin Bridge                  Rust SDK
      │                                    │                            │
      │─── postMessage([msgId][req]) ─────►│                            │
      │                                    │── JNI call ───────────────►│
      │                                    │                            │  process
      │                                    │◄── protobuf result ────────│
      │◄── postMessage([msgId][resp]) ─────│                            │
      │                                    │                            │
```

### Timeout

The bridge timeout is 30000ms (30 seconds). Configured in both:
- `dsm_client/new_frontend/public/index.html`
- `dsm_client/android/app/src/main/assets/index.html`

## JNI Interface

The Rust SDK exposes 87+ JNI methods to the Android runtime. All methods are declared in `UnifiedNativeApi.kt` and implemented in `unified_protobuf_bridge.rs`.

### JNI Method Categories

| Category | Methods | Description |
|----------|---------|-------------|
| Bootstrap | `sdkBootstrap`, `sdkReady` | Initialize SDK, check readiness |
| Genesis | `createGenesis`, `extractGenesisIdentity` | MPC genesis creation |
| Wallet | `getBalance`, `getTransactions`, `getDeviceId` | Balance and identity queries |
| Bilateral | `bilateralPrepare`, `bilateralAccept`, `bilateralCommit` | 3-phase bilateral protocol |
| BLE | `bleCommand`, `bleStartScan`, `bleStopScan` | BLE operations |
| DLV | `dlvCreate`, `dlvUnlock`, `dlvStatus` | Vault management |
| Bitcoin | `bitcoinGenerateAddress`, `bitcoinSwapInitiate`, `bitcoinSwapComplete` | dBTC bridge |
| Token | `tokenCreate`, `tokenTransfer`, `tokenBurn` | Token operations |
| Storage | `storagePut`, `storageGet`, `storageSync` | Storage node interaction |
| Envelope | `processEnvelopeV3` | Generic envelope processing |

### JNI Symbol Convention

All JNI symbols follow the pattern:

```
Java_com_dsm_wallet_bridge_UnifiedNativeApi_<methodName>
```

Verify symbol count after building:

```bash
nm -gU libdsm_sdk.so | grep -c Java_
# Expected: 87+
```

### JNI Data Exchange

All JNI methods accept and return **protobuf-encoded byte arrays** (`jbyteArray`). The Kotlin side converts between Java byte arrays and the protobuf message types.

```kotlin
// Kotlin side (UnifiedNativeApi.kt)
external fun getBalance(request: ByteArray): ByteArray

// Rust side (unified_protobuf_bridge.rs)
#[no_mangle]
pub extern "system" fn Java_com_dsm_wallet_bridge_UnifiedNativeApi_getBalance(
    env: JNIEnv,
    _class: JClass,
    request: jbyteArray,
) -> jbyteArray {
    // Decode protobuf request, process, encode protobuf response
}
```

## Protobuf Schema

The canonical schema lives at `proto/dsm_app.proto` (schema v2.4.0, wire v3).

### Key Message Types

| Message | Purpose |
|---------|---------|
| `Envelope` | Wire container for all messages |
| `Hash32` | 32-byte hash (BLAKE3) |
| `BridgeRpcRequest` / `BridgeRpcResponse` | Bridge communication |
| `BilateralPrepare` / `BilateralAccept` / `BilateralCommit` | 3-phase bilateral |
| `DlvCreate` / `DlvUnlock` | DLV vault operations |
| `AtomicSwapInitiate` / `AtomicSwapComplete` | Bitcoin bridge |
| `BleCommand` / `BleCommandResponse` | BLE operations |
| `Evidence` | Proof data for storage nodes |
| `ByteCommit` | State commitment for anchoring |

### Proto Regeneration

After editing `proto/dsm_app.proto`:

```bash
cd dsm_client/new_frontend
npm run proto:gen
```

This generates `src/proto/dsm_app_pb.ts` using `@bufbuild/protoc-gen-es`. Always regenerate — never inline or duck-type proto shapes.

### Proto Schema Versioning

- Schema version: 2.4.0 (in the `.proto` file header)
- Wire version: 3 (Envelope v3)
- Breaking changes require a schema version bump

## BLE Command Protocol

BLE operations use a dedicated command/response pattern routed through the SDK:

```
Frontend → BleCommand (protobuf) → SDK → BLE backend → BleCommandResponse
```

The SDK accepts an Envelope v3 `UniversalTx` carrying an `ArgPack { codec=PROTO, body=BleCommand }`, dispatches to the registered BLE backend, and returns a `ResultPack { codec=PROTO, body=BleCommandResponse }`.

### BLE Transport Chunking

BLE has MTU limitations (typically 20–512 bytes per packet). The `ble_frame_coordinator.rs` handles:

1. **Segmentation:** Large protobuf messages are split into MTU-sized frames
2. **Reassembly:** Frames are collected and reassembled on the receiving end
3. **Ordering:** Sequence numbers ensure correct ordering
4. **Retransmission:** Lost frames can be retransmitted

### Radio Timing

A 500ms gap is required between scan/advertise switches to avoid BLE radio state conflicts. The `BleCoordinator` in Kotlin enforces this constraint using an actor pattern with Channel-serialized operations.

## Cross-Boundary Endpoint Matrix

Complete inventory of every endpoint that crosses JS ↔ Android ↔ Rust boundaries.

**Invariant:** protocol traffic is raw bytes only across boundaries. Base32 Crockford is UI/debug-only.

### Common Framing Helpers (JS)

- `u32be(n)` = 4-byte big-endian length
- `frameCall(name, payload)` = `[u32be(nameLen)][nameUtf8][payloadBytes]`
- Router correlation prefix: `appRouterInvoke` + `appRouterQuery` prefix **8-byte** `requestId` to payload, strip 8 bytes from response
